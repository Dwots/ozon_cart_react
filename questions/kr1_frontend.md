## 0. Введение: как вообще устроен этот проект (прочитай первым)

Представь, что страница сайта — это **конструктор из кубиков**. Каждый кубик — это **компонент**. Из маленьких кубиков собираются большие.

В нашем проекте кубики такие:
- `App` — самый главный кубик, внутри него лежат все остальные.
- `Header` — шапка сверху (логотип, статусы, кнопка «Корзина»).
- `ProductsPage` — страница со списком товаров.
- `AddToCartButton` — кнопка «Добавить в корзину» (она повторяется у каждого товара).
- `CartDrawer` — выезжающая сбоку панель корзины.

Каждый компонент — это **обычная JS-функция**, которая возвращает разметку (похожую на HTML). Вот самый простой пример из проекта:

```jsx
function AddToCartButton({ product, onAdd, disabled = false }) {
  return (
    <button type='button' onClick={() => onAdd(product)} disabled={disabled}>
      {disabled ? 'Нет в наличии' : 'Добавить в корзину'}
    </button>
  );
}
```

Это функция `AddToCartButton`. Она возвращает кнопку. React берёт то, что вернула функция, и рисует это на экране. Всё. Компонент = функция, которая рисует кусок страницы.

Эта «разметка внутри JS» называется **JSX**.

Дальше идут вопросы. Если что-то непонятно — возвращайся сюда.

## 1. React и компоненты — основы

### 1. Что такое React и его основная идея?

**На пальцах:** обычно, чтобы поменять что-то на странице, надо вручную найти элемент и переписать его (`document.getElementById...`). Это муторно. React делает по-другому: ты просто говоришь «вот так должна выглядеть страница при таких данных», а React сам меняет нужное.

**Идея:** ты описываешь *результат* (что показать), а не *шаги* (как менять). Это называется **декларативность**. Плюс страница собирается из переиспользуемых **компонентов** (кубиков).

### 2. Что такое компонент? Функциональный vs классовый?

**На пальцах:** компонент — это кубик страницы, оформленный как функция.

```jsx
function Header({ totalCount }) {
  return <header>Корзина: {totalCount}</header>;
}
```

**Разбор:** `Header` — компонент. Раньше в старом React компоненты писали как классы (`class Header extends ...`) — это «классовый». Сейчас пишут функциями — это «функциональный». **У нас все компоненты функциональные** — так проще, и только с ними работают хуки.

### 3. Что такое JSX и во что компилируется?

**На пальцах:** JSX — это когда мы пишем «HTML» прямо внутри JavaScript.

```jsx
return <h2>Товары</h2>;
```

**Разбор:** браузер такое не понимает — для него `<h2>` посреди JS это ошибка. Поэтому перед запуском специальная программа (**Babel**) превращает JSX в обычный JS-код вида `React.createElement('h2', null, 'Товары')`. Мы пишем удобный JSX, а на выходе получается понятный браузеру код.

### 4. Virtual DOM и реконсиляция?

**На пальцах:** DOM (реальная страница) — медленная штука, часто её менять дорого. Поэтому React держит **копию страницы в памяти** (Virtual DOM, «черновик»).

**Разбор:** когда данные меняются, React рисует новый черновик, **сравнивает** его со старым (этот процесс — *реконсиляция*) и меняет на реальной странице **только то, что реально изменилось**. Например, если поменялось число в корзине — обновится только это число, а не вся страница.

### 5. Что такое рендер? Когда происходит?

**На пальцах:** «рендер» = React вызвал функцию компонента и получил свежую разметку.

**Разбор:** рендер происходит: (1) когда компонент впервые появляется на экране и (2) когда у него меняются данные — **state** или **props** (про них ниже). После рендера React обновляет страницу.

### 6. Почему компонент пишут с большой буквы?

```jsx
<Header />      // наш компонент (с большой буквы)
<div></div>     // обычный html-тег (с маленькой)
```

**Разбор:** так React отличает свои компоненты от обычных HTML-тегов. Если написать `<header>` с маленькой — React подумает, что это просто HTML-тег, а не твой компонент `Header`.

### 7. Можно ли вернуть два элемента без обёртки? Fragment?

**На пальцах:** функция-компонент должна вернуть **один** элемент. Если надо вернуть два рядом — их нужно во что-то завернуть. Чтобы не плодить лишние `<div>`, есть «пустая обёртка» — **Fragment**: `<>...</>`.

```jsx
return (
  <>
    <div className="cart-overlay" />
    <aside className="cart-drawer">...</aside>
  </>
);
```

**Разбор (`CartDrawer`):** тут возвращаются два элемента — затемнение (`overlay`) и сама панель (`aside`). Они обёрнуты в `<>...</>` — это Fragment. Он группирует их, но **сам не создаёт лишнего тега** в HTML.

## 2. Props — передача данных в компонент

### 8. Что такое props и куда передаются?

**На пальцах:** props — это «настройки», которые **родитель передаёт ребёнку**. Как параметры функции. Поток всегда сверху вниз: родитель → ребёнок.

```jsx
// App (родитель) передаёт Header'у число totalCount:
<Header totalCount={totalCount} />
```

**Разбор:** `App` говорит `Header`: «вот тебе число `totalCount`, нарисуй его». `Header` получает это как props.

### 9. Деструктуризация props?

**На пальцах:** все props приходят в один объект. Чтобы не писать `props.totalCount`, `props.isOnline` каждый раз, мы сразу «разбираем» объект на отдельные переменные прямо в скобках.

```jsx
function Header({ totalCount, isOnline, workerStatus, onCartOpen }) {
```

**Разбор:** вместо `function Header(props)` и потом `props.totalCount` — мы сразу достали `totalCount`, `isOnline` и т.д. Это и есть деструктуризация. Чисто удобство записи.

### 10. Значение по умолчанию для prop?

```jsx
function AddToCartButton({ product, onAdd, disabled = false }) {
```

**Разбор:** `disabled = false` значит: если родитель **не передал** `disabled`, считаем, что он `false` (кнопка активна). Это страховка, чтобы не было `undefined`.

### 11. Можно ли менять props внутри компонента?

**На пальцах:** нет. Props — это как чужая посылка: ты её получил, можешь посмотреть, но переписывать чужое нельзя.

**Разбор:** компонент **не должен** менять свои props. Если ему нужно что-то менять — он либо сообщает об этом родителю (через функцию-проп), либо использует свой собственный **state**.

### 12. Зачем передавать функции через props?

**На пальцах:** иногда родитель даёт ребёнку не данные, а **кнопку-действие**: «нажми сюда, когда надо что-то сделать».

```jsx
// App передаёт функцию addToCart вниз:
<Cart addToCart={addToCart} />
```

**Разбор:** логика «добавить в корзину» живёт в `App`. Но кнопка находится в товаре далеко внизу. Поэтому `App` передаёт саму функцию вниз, и ребёнок её вызывает, когда юзер кликнул.

### 13. Callback-пропсы (onClose, onCartOpen)?

**На пальцах:** это способ для ребёнка «крикнуть наверх» родителю: «эй, на меня нажали!».

```jsx
// В App:
<Header onCartOpen={() => setIsCartOpen(true)} />

// Внутри Header:
<button onClick={onCartOpen}>Корзина</button>
```

**Разбор:** `Header` сам не умеет открывать корзину. Но когда на кнопку нажали, он вызывает `onCartOpen` — а это функция из `App`, которая реально открывает корзину. То есть ребёнок сообщает о событии, а действие выполняет родитель.

## 3. State — собственная «память» компонента

### 14. State vs props? (с примером!)

**На пальцах:**
- **props** — то, что тебе **дали снаружи** (как зарплата от работодателя — сам себе не меняешь).
- **state** — твои **личные данные**, которые ты сам меняешь (как заметки в своём блокноте).

**Пример props** (`Header` получил число снаружи и не меняет его):
```jsx
function Header({ totalCount }) {   // totalCount пришёл от App — это props
  return <span>{totalCount}</span>;
}
```

**Пример state** (`App` хранит и сам меняет — открыта корзина или нет):
```jsx
const [isCartOpen, setIsCartOpen] = useState(false);  // это state
// ...
<button onClick={() => setIsCartOpen(true)}>Открыть</button>  // App сам меняет своё состояние
```

**Разбор:** `totalCount` в `Header` — это props (пришёл сверху, Header его не трогает). `isCartOpen` в `App` — это state (App сам его создал и сам меняет через `setIsCartOpen`).

### 15. Что делает useState?

**На пальцах:** даёт компоненту **память** + **кнопку для её изменения**.

```jsx
const [isCartOpen, setIsCartOpen] = useState(false);
//      ↑значение      ↑функция-изменятель   ↑стартовое значение
```

**Разбор:** `useState(false)` возвращает массив из двух штук:
1. `isCartOpen` — текущее значение (сейчас `false`).
2. `setIsCartOpen` — функция, которой это значение меняют.

Когда вызовешь `setIsCartOpen(true)` — значение станет `true` и компонент перерисуется.

### 16. Почему нельзя менять state напрямую?

**На пальцах:** если поменять «втихую», React не заметит и не перерисует экран.

```jsx
// ПЛОХО — React не узнает об изменении:
cart.push(newItem);

// ХОРОШО — через сеттер, React перерисует:
setCart([...cart, newItem]);
```

**Разбор:** менять надо **только через сеттер** (`setCart`). Иначе данные поменяются, а на экране останется старое.

### 17. Ленивая инициализация `useState(getSavedCart)`?

**На пальцах:** мы хотим при старте подгрузить корзину из памяти браузера. Но читать память на каждом обновлении — расточительно. Поэтому передаём **саму функцию**, и React вызовет её **только один раз** при создании.

```jsx
const [cart, setCart] = useState(getSavedCart);   // правильно: передаём функцию
// а НЕ useState(getSavedCart())                   // это бы выполнялось каждый рендер
```

**Разбор:** `getSavedCart` (без скобок) — это «рецепт, как получить начальное значение». React выполнит его один раз при первом рендере. Если бы написали `getSavedCart()` со скобками — оно бы пересчитывалось зря каждый раз.

### 18. Что происходит при вызове сеттера?

**Разбор:** вызвал `setCart(...)` → React понимает «данные изменились» → заново вызывает функцию компонента (рендер) → обновляет экран. Сеттер — это и есть «сигнал перерисоваться».

### 19. Функциональная форма `setCart(prev => ...)`?

**На пальцах:** когда новое значение зависит от старого, безопаснее не брать старое «снаружи», а попросить React дать актуальное.

```jsx
setCart((prevCart) => {
  const nextCart = [...prevCart, newItem];  // берём актуальный prevCart
  return nextCart;
});
```

**Разбор:** `prevCart` — это гарантированно **самое свежее** значение корзины. Так не наткнёшься на ситуацию, когда несколько изменений подряд перетёрли друг друга.

### 20. Где хранится `isCartOpen` и почему в App, а не в CartDrawer?

```jsx
// в App.js:
const [isCartOpen, setIsCartOpen] = useState(false);
```

**На пальцах:** открытие корзины касается двоих: **кнопки в шапке** (`Header`) и **самой панели** (`CartDrawer`). Если бы это состояние было внутри `CartDrawer`, шапка не смогла бы им управлять.

**Разбор:** поэтому состояние держат в `App` — общем «начальнике» обоих. Это называется «поднять состояние наверх» (см. блок 4).

## 4. Поднятие состояния и архитектура

### 21. Lifting state up (поднятие состояния)?

**На пальцах:** если двум кубикам нужны одни и те же данные — кладём данные в их **общего родителя**, а оттуда раздаём вниз.

**Разбор:** вся корзина живёт в хуке `useCart`, который вызван в `App`. И `Header`, и `CartDrawer` получают данные из `App`. Так у обоих всегда **одинаковые** цифры.

### 22. Как Header и CartDrawer всегда показывают одно и то же?

```jsx
const { totalCount, ... } = useCart();   // один источник данных в App
<Header totalCount={totalCount} />
<CartDrawer totalCount={totalCount} ... />
```

**Разбор:** оба берут `totalCount` из **одного места**. Поменялась корзина → поменялось `totalCount` → обновились **оба** компонента сразу. Рассинхрона быть не может.

### 23. Почему totalCount считается в хуке, а не в каждом компоненте?

**Разбор:** чтобы посчитать количество товаров **один раз** в `useCart` и отдать готовое число и в шапку, и в корзину. Иначе пришлось бы дублировать одну и ту же формулу в двух местах.

## 5. Хуки — что это вообще такое

### 24. Что такое хук и зачем? (с примером!)

**На пальцах:** хук — это **готовый инструмент**, который компонент «берёт в руки», чтобы что-то уметь. Сам по себе компонент — просто функция, рисующая разметку. Хуки добавляют ему способности: помнить данные, реагировать на события, и т.д.

**Как узнать хук:** его имя **начинается со слова `use`** — `useState`, `useEffect`, `useRef`. «Функция вида useXxx» означает просто «функция, чьё имя начинается на use».

```jsx
const [cart, setCart] = useState([]);   // useState — хук «дай мне память»
useEffect(() => { ... }, []);           // useEffect — хук «сделай что-то после рендера»
```

**Разбор:** в `useState([])` мы взяли инструмент «память» — теперь компонент помнит корзину. Без хуков функция-компонент ничего не помнит и ни на что не реагирует — просто рисует.

### 25. Правила хуков (почему нельзя в if/циклах)?

**На пальцах:** React запоминает хуки **по порядку** их вызова — первый, второй, третий. Если спрятать хук в `if`, то иногда он вызовется, иногда нет — и порядок «поедет», React перепутает, где чья память.

```jsx
// ПЛОХО:
if (isOpen) {
  const [x, setX] = useState(0);   // нельзя! иногда есть, иногда нет
}

// ХОРОШО — всегда на верхнем уровне:
const [x, setX] = useState(0);
```

**Разбор:** поэтому хуки вызывают **всегда вверху компонента, всегда в одном порядке**, без условий и циклов.

### 26. Откуда можно вызывать хуки? (с примером!)

**На пальцах:** хуки работают **только внутри компонента** (или внутри другого хука). В обычной функции, не связанной с React, они работать не будут — React не поймёт, чьи это данные.

```jsx
// МОЖНО — внутри компонента:
function App() {
  const [isOpen, setIsOpen] = useState(false);   // ок, мы внутри компонента
  ...
}

// МОЖНО — внутри своего хука:
export function useCart() {
  const [cart, setCart] = useState([]);          // ок, useCart — это хук
  ...
}

// НЕЛЬЗЯ — в обычной функции:
function saveCart(cart) {
  const [x, setX] = useState(0);   // ошибка! saveCart — не компонент и не хук
}
```

**Разбор:** `App` — компонент, `useCart` — хук (имя на `use`) → внутри них хуки разрешены. А `saveCart` — просто функция-помощник для записи в localStorage, она не часть React → хуки в ней запрещены.

## 6. useEffect — «сделай что-то после рендера»

### 27. Что такое useEffect?

**На пальцах:** иногда после того, как компонент нарисовался, надо сделать что-то «снаружи» React: запросить данные с сервера, подписаться на события, завести таймер. Для этого есть `useEffect`.

```jsx
useEffect(() => {
  navigator.serviceWorker.register('/service-worker.js');
}, []);
```

**Разбор:** «нарисовался компонент → выполни вот этот код». Здесь — зарегистрируй Service Worker.

### 28. Массив зависимостей `[]`?

**На пальцах:** второй аргумент `useEffect` — список «за чем следить». От него зависит, **как часто** эффект повторяется.

```jsx
useEffect(() => { ... }, []);          // [] — один раз при появлении компонента
useEffect(() => { ... }, [cart]);      // [cart] — каждый раз когда меняется cart
useEffect(() => { ... });              // без массива — после каждого рендера
```

**Разбор:** пустой `[]` = «сделай один раз при старте и забудь». Именно так у нас подключается SharedWorker — один раз.

### 29. Функция очистки (return внутри useEffect)?

**На пальцах:** если эффект что-то «включил» (например, подписался на событие), надо уметь это «выключить», когда компонент исчезает. Для этого эффект может вернуть функцию-уборщика.

```jsx
useEffect(() => {
  window.addEventListener('online', handleOnline);    // подписались

  return () => {
    window.removeEventListener('online', handleOnline); // отписались (уборка)
  };
}, []);
```

**Разбор:** код до `return` — «включаем» (подписка на событие «появился интернет»). Функция в `return` — «выключаем» (отписка). React вызовет её, когда компонент уберут с экрана. Так не остаётся «висящих» подписок.

### 30. Зачем флаг `cancelled` в useProducts?

**На пальцах:** мы запросили товары с сервера, но это медленно. Вдруг пользователь ушёл со страницы раньше, чем пришёл ответ? Тогда обновлять компонент уже нельзя — его нет. Флаг `cancelled` это ловит.

```jsx
let cancelled = false;
async function loadProducts() {
  const data = await fetch(...);          // долгий запрос
  if (!cancelled) {                       // компонент ещё на экране?
    setProducts(data);                    // только тогда обновляем
  }
}
return () => { cancelled = true; };       // ушли со страницы → ставим флаг
```

**Разбор:** если компонент убрали, уборщик ставит `cancelled = true`, и когда ответ всё же придёт — мы его проигнорируем. Без этого React ругался бы «обновляешь то, чего уже нет».

### 31. Почему Service Worker регистрируют в useEffect, а не просто в теле?

**Разбор:** регистрация SW — это «побочное действие» (лезем в браузерный API). Такое нельзя делать прямо в теле компонента, потому что тело вызывается **на каждый рендер** — получили бы кучу повторных регистраций. В `useEffect` с `[]` оно выполнится аккуратно **один раз** после первого рендера.

### 32. Сколько useEffect в useCart и за что отвечают?

**Разбор:** два эффекта:
1. **Подключение к SharedWorker** — создаёт воркер, слушает сообщения о корзине.
2. **Слежение за интернетом** — подписка на события `online`/`offline`, чтобы показывать статус сети.

Их разделили, потому что это **два разных дела** — мухи отдельно, котлеты отдельно.

## 7. useMemo и useRef

### 33. Что делает useMemo?

**На пальцах:** «не пересчитывай заново, если входные данные не менялись — запомни прошлый результат».

```jsx
const totalPrice = useMemo(() => {
  return cart.reduce((sum, item) => sum + item.price * item.quantity, 0);
}, [cart]);
```

**Разбор:** считаем общую сумму корзины. `useMemo` с `[cart]` означает: пересчитывай сумму **только когда меняется `cart`**. Если перерисовка случилась по другой причине — берём готовый прошлый результат, не считаем зря.

### 34. Что было бы без useMemo?

**Разбор:** всё работало бы так же — просто сумма пересчитывалась бы на **каждом** рендере, даже когда корзина не менялась. Это про **оптимизацию**, а не про «сломается/не сломается».

### 35. Что такое useRef? Отличие от useState?

**На пальцах:** `useRef` — это тоже «память», но **тихая**: когда меняешь её, экран **не** перерисовывается. `useState` при изменении — перерисовывает.

```jsx
const portRef = useRef(null);   // тихая память
portRef.current = port;         // поменяли — экран НЕ перерисовался
```

**Разбор:** используем ref для вещей, которые **не нужно показывать** на экране, но надо хранить между рендерами.

### 36. Почему порт SharedWorker в portRef, а не в state? (важный вопрос)

**На пальцах:** порт — это «телефонная линия» к воркеру, техническая штука. Её не рисуют на экране, и менять её незачем с перерисовкой.

```jsx
const portRef = useRef(null);
portRef.current = port;                       // храним связь
// потом из любой функции:
portRef.current.postMessage({ type: 'ADD_ITEM', product });   // используем
```

**Разбор:** если бы хранили порт в `useState`, любое присвоение дёргало бы лишнюю перерисовку без пользы. А ещё `ref` удобно доступен из всех функций хука (`addToCart`, `removeFromCart` и т.д.).

## 8. Кастомные (свои) хуки

### 37. Что такое кастомный хук?

**На пальцах:** это **твоя собственная** функция-инструмент, в которую ты спрятал кусок логики, чтобы переиспользовать. Главное правило — имя на `use`.

```jsx
export function useCart() {   // наш хук
  const [cart, setCart] = useState(getSavedCart);
  ...
  return { cart, totalCount, addToCart, ... };
}
```

**Разбор:** `useCart` — это мы сами написали. Внутри он пользуется встроенными хуками (`useState`, `useEffect`...) и отдаёт наружу готовую корзину и функции.

### 38. Зачем useCart / useProducts вместо кода прямо в компонентах?

**На пальцах:** чтобы компонент занимался только **рисованием**, а вся «кухня» (логика корзины, загрузка товаров) лежала отдельно и аккуратно.

**Разбор:** `App` не захламлён деталями работы с воркером и localStorage — он просто пишет `const { cart, addToCart } = useCart()` и пользуется готовым.

### 39. Что возвращает useCart?

```jsx
return {
  cart, totalCount, totalPrice, isOnline, workerStatus,
  addToCart, removeFromCart, decreaseItem, clearCart,
};
```

**Разбор:** объект с **данными** (корзина, кол-во, сумма, статусы) и **функциями-действиями** (добавить, удалить, уменьшить, очистить). `App` достаёт нужное через деструктуризацию.

### 40. Может ли хук использовать другие хуки?

**Разбор:** да, в этом вся суть. `useCart` внутри использует `useState`, `useEffect`, `useMemo`, `useRef` все сразу. Хуки складываются друг в друга как кубики.

## 9. Списки и условный рендеринг

### 41. Зачем `key` при выводе списка?

**На пальцах:** когда рисуем список, React должен понимать «кто есть кто», чтобы при изменениях не перепутать элементы. `key` — это уникальная бирка на каждом элементе.

```jsx
{cart.map((item) => (
  <div className="cart-drawer-card" key={item.id}>...</div>
))}
```

**Разбор:** `key={item.id}` помогает React точечно обновлять список. Без `key` он ругается и может обновлять не то.

### 42. Почему key = product.id, а не индекс?

**На пальцах:** `id` у товара **уникальный и постоянный**. А индекс (0,1,2...) меняется, если элемент удалить или переставить — и бирки «съедут».

**Разбор:** поэтому ключом берут стабильный `id`, а не позицию в массиве.

### 43. Условный рендеринг через тернарник?

**На пальцах:** «если корзина пустая — покажи одно, иначе — другое».

```jsx
{cart.length === 0 ? (
  <p className="cart-empty">Корзина пустая</p>
) : (
  <div className="cart-drawer-list">...список товаров...</div>
)}
```

**Разбор:** `условие ? А : Б` — рисует `А`, если условие истинно, иначе `Б`. Здесь по пустоте корзины выбираем, что показать.

### 44. Что делает `totalCount > 0 && <span>`?

**На пальцах:** «покажи бейдж с числом, только если в корзине что-то есть».

```jsx
{totalCount > 0 && <span className="cart-count">{totalCount}</span>}
```

**Разбор:** оператор `&&` работает так: если слева `true` — покажи то, что справа; если слева `false` — не показывай ничего. При `totalCount = 0` бейдж не рисуется.

### 45. Статусы загрузки в ProductsPage?

```jsx
{status === 'loading' && <p>Загрузка товаров…</p>}
{status === 'fallback' && <p>Показан сохранённый список.</p>}
```

**Разбор:** пока товары грузятся — `status` равен `'loading'`, показываем «Загрузка…». Если загрузка не удалась и взяли резервные данные — `status` равен `'fallback'`, показываем предупреждение. Иначе просто рисуется список.

### 46. Что делает rating.toFixed(1)?

```jsx
{product.rating.toFixed(1)}    // 4.8 → "4.8", а 5 → "5.0"
```

**Разбор:** округляет число до **одного знака после запятой**, чтобы рейтинг выглядел ровно.

## 10. События (клики и т.п.)

### 47. `onClick={onClose}` vs `onClick={() => onDecrease(item.id)}`?

**На пальцах:**
- Если **не надо** передавать аргумент — даём саму функцию: `onClick={onClose}`.
- Если **надо** передать аргумент — оборачиваем в стрелку: `onClick={() => onDecrease(item.id)}`.

```jsx
<button onClick={onClose}>Закрыть</button>                       // просто вызвать
<button onClick={() => onDecrease(item.id)}>-</button>           // вызвать с аргументом
```

**Разбор:** во втором случае стрелка нужна, чтобы функция **не вызвалась сразу при рисовании**, а дождалась клика и тогда передала `item.id`.

### 48. Что будет от `onClick={onDecrease(item.id)}` без стрелки? (подвох)

**Разбор:** функция вызовется **сразу во время рендера**, а не по клику. То есть товары будут «уменьшаться» сами при отрисовке. Это ошибка. Поэтому и нужна стрелка `() => ...`.

### 49. Кнопка «+»: почему передаём весь item, а не item.id?

```jsx
<button onClick={() => onAdd(item)}>+</button>
```

**Разбор:** функция `addToCart` рассчитана на **целый объект товара** (вдруг товара ещё нет в корзине — его надо добавить со всеми полями: название, цена...). У `item` все поля уже есть, поэтому передаём его целиком.

## 11. UI-механика и стили

### 50. Как открывается панель корзины без удаления из DOM?

**На пальцах:** панель **всегда** есть на странице, просто она «уехала» за край. При открытии добавляем CSS-класс, и она выезжает (анимация).

```jsx
<aside className={`cart-drawer ${isOpen ? 'cart-drawer-open' : ''}`}>
```

**Разбор:** если `isOpen` истина — к классу добавляется `cart-drawer-open` (по нему CSS показывает панель). Если ложь — класса нет, панель спрятана. Сам элемент из страницы не удаляется.

### 51. Зачем `aria-hidden={!isOpen}`?

**Разбор:** это для **доступности** (программы для незрячих — скринридеры). Когда панель закрыта (`isOpen = false`), `aria-hidden` становится `true` — скринридер «не читает» спрятанную панель, чтобы не путать пользователя.

### 52. Как работает overlay (затемнение)?

```jsx
<div className="cart-overlay ..." onClick={onClose} />
```

**Разбор:** это тёмный полупрозрачный слой поверх страницы, когда корзина открыта. Клик по нему вызывает `onClose` — удобный способ закрыть корзину, нажав «мимо».

### 53. Как работает `import './App.css'`?

**Разбор:** в проектах на Create React App можно **импортировать CSS прямо в JS**. Сборщик увидит этот импорт и подключит стили к странице. Так стили из `App.css` начинают действовать.

## 12. Роутинг (переключение «страниц»)

### 54. Зачем роутер, если страница одна?

**Разбор:** роутер (`react-router-dom`) управляет тем, что показать по разным адресам (`/cart`). Здесь он используется для редиректов и задела на будущее — даже при одной странице это правильная заготовка.

### 55. Routes / Route / Navigate?

```jsx
<Routes>
  <Route path="/" element={<Navigate to="/cart" replace />} />
  <Route path="/cart" element={<Cart addToCart={addToCart} />} />
  <Route path="*" element={<Navigate to="/cart" replace />} />
</Routes>
```

**Разбор:**
- `<Routes>` — контейнер со списком маршрутов.
- `<Route path="..." element={...} />` — один маршрут: «по такому адресу показывай такой компонент».
- `<Navigate to="/cart" />` — редирект (перекинуть на `/cart`).

Здесь: адрес `/` перекидывает на `/cart`.

### 56. Что делает `path="*"`?

**Разбор:** звёздочка ловит **любой несуществующий адрес**. Зашёл на `/abrakadabra` → перекинет на `/cart`. Защита от «страница не найдена».

### 57. Link vs обычный `<a href>`?

```jsx
<Link to="/cart">Ozon Cart</Link>
```

**На пальцах:** обычная ссылка `<a>` **перезагружает всю страницу**. `<Link>` меняет адрес **без перезагрузки** — приложение остаётся «живым», просто показывает другой маршрут. Это и есть SPA (одностраничное приложение).

### 58. Зачем BrowserRouter в index.js?

```jsx
<BrowserRouter>
  <App />
</BrowserRouter>
```

**Разбор:** чтобы `Routes`, `Route`, `Link` внутри `App` вообще работали, всё приложение нужно обернуть в `BrowserRouter`. Он «включает» механизм маршрутизации.

## 13. Точка входа (с чего стартует приложение)

### 59. Что делает index.js?

```jsx
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<BrowserRouter><App /></BrowserRouter>);
```

**Разбор:** это **самый первый** запускаемый файл. Он находит на странице место (`#root`) и говорит React: «нарисуй здесь мой `App`». С этого момента React берёт управление.

### 60. Куда монтируется приложение?

**Разбор:** в пустой `<div id="root">` внутри `public/index.html`. Весь сайт React рисует внутри этого одного div.

## 14. Каверзные / на понимание

### 61. Почему корзина не теряется при перезагрузке страницы?

**На пальцах:** корзину сохраняем в память браузера (**localStorage**), а при старте читаем её обратно.

```jsx
const [cart, setCart] = useState(getSavedCart);   // при старте читаем из localStorage
// ... и при каждом изменении:
saveCart(nextCart);                                // сохраняем обратно
```

**Разбор:** даже если закрыть вкладку и открыть заново — `getSavedCart` достанет сохранённую корзину из localStorage.

### 62. Опиши порядок работы useProducts.

**Разбор (по шагам):**
1. Компонент рисуется первый раз, `status = 'loading'` → видим «Загрузка…».
2. После рендера срабатывает `useEffect`.
3. Внутри идёт `fetch` — запрос за товарами.
4. Пришли данные → `setProducts(data)` + `status = 'success'`.
5. Изменение state → перерисовка → теперь виден список товаров.

### 63. Что если fetch товаров упал?

```jsx
} catch {
  setProducts(fallbackProducts);   // берём встроенный список
  setStatus('fallback');
}
```

**Разбор:** если запрос не удался, берём **резервный список** из `data/products.js`, ставим статус `'fallback'` и показываем сообщение. Каталог не останется пустым.

### 64. Почему товар с stock=0 не кликается?

```jsx
const inStock = product.stock > 0;
<AddToCartButton product={product} onAdd={addToCart} disabled={!inStock} />
```

**Разбор:** если на складе 0 (`inStock = false`), кнопке передаётся `disabled={true}` — она становится неактивной и показывает «Нет в наличии».

### 65. Что делает `inStock = product.stock > 0`?

**Разбор:** это просто флаг «есть ли товар на складе» (`true`/`false`). От него зависит: текст наличия, CSS-класс (`product-stock-out` для красного цвета) и активность кнопки.

### 66. Зачем `typeof navigator !== 'undefined'`?

```jsx
const [isOnline, setIsOnline] = useState(
  typeof navigator !== 'undefined' ? navigator.onLine : true
);
```

**Разбор:** в редких случаях (например, если код запускается не в браузере) объекта `navigator` может не быть. Эта проверка — страховка, чтобы код не упал с ошибкой «navigator не определён».
